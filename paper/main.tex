\abstract{
  The contract data visualization tools make with their users is that a chart is a faithful and accurate visual representation of the numbers it is made from. Motivated by wanting to make tools that are more flexible while still rigorously structure preserving, we propose a methodology for fully specifying arbitrary data to visualization mappings in a manner that translates to pseudocode. We propose that fiber bundles provide a uniform interface for describing a variety of underlying data - tables, images, networks, etc. - in a manner that independently encodes the mathematical structure of the topology and the fields of the dataset. Modeling the data structures that store the datasets as sheaves provides a framework for specifying visualization methods that are designed to work regardless of how the dataset is stored - whether the data is on disk, distributed, or on demand. Specifying the visualization library components as natural transforms of sheaves means that the constraints that the component must satisfy to be structure preserving can be specified as the set of morphisms on the data and graphic sheaves, including the structure on the topology and fields of the data. Using category theory to formally express how visual elements are constructed means we can translate those expectations into pseudocode, which can then be used to build tools that enforce the expectation that a visualization tool is faithfully translating between numbers and charts.
}

\firstsection{Introduction}\label{sec:intro}

\maketitle

Visualization design guidelines, generally, describe how to choose data $\rightarrow$ graphic encodings that preserve the structure of the data;  Loosely, preserving structure means that the properties of the data and how the points are connected to each other should be inferable from the graphic such that a graphic $\rightarrow$ data mapping can be made. For example, values read off a bar chart have to be equivalent to the values used to construct that chart. Therefore, a visualization tool is structure preserving when it preserves the bidirectional mapping data $\leftrightarrow$ graphic. We propose that structures from category theory and algebraic topology can be used as a sort of pseudocode for uniformly specifying the behaviors and properties of data and graphic and rigorously expressing the structure preserving constraints on maps between them. By design, this pseudocode translates to strong programmatic types and a functional design framework, which means these components can be built as a standalone library and integrated into existing libraries. The contribution of this paper is a methodology for describing structure, verifying structure preservation, and specifying the conditions for constructing a structure preserving map between data and graphics. This framework also provides guidance for the construction and testing of structure preserving visualization library components.
\section{Related Work}
\label{sec:related-work}
\label{sec:related-work:equivariance}
Our model generalizes previous mathematically derived frameworks by allowing arbitrary structures, explicitly incorporating topology, and by providing an explicit framework for translating the theoretical ideas into buildable components. Bertin\cite{bertinSemiologyGraphicsDiagrams2011} codified the idea that properties of data and graphics should match, using the Steven's measurement scales \cite{stevensTheoryScalesMeasurement1946} as a case study. Mackinlay and Wilkinson translate algebras on data to grammars for specifying graphics to develop frameworks for designing structure preserving visualization tools \cite{mackinlayAutomatingDesignGraphical1986,mackinlayAutomaticDesignGraphical1987, wilkinsonGrammarGraphics2005}, while our framework uses the same abstractions for data and graphics. Vickers et al \cite{vickersUnderstandingVisualizationFormal2013} apply category theory to visualization semantics, while our focus is structural. Bartonicek built a tool based on the algebraic properties of summary statistics \cite{bartonicekNoMoreNo2025}; our work provides a more general framework for codifying those properties and proposes that sheaves are a better abstraction for reasoning about connectivity constraints. Similarly to Kindlemann and Scheidegger's algebraic visualization design (AVD) framework \cite{kindlmannAlgebraicProcessVisualization2014} --- but with a focus on software architecture rather than visualization design --- we present a uniform abstract data representation layer for ensuring that the visualization should not change if the data representation changes, define the conditions under which data is mapped unambiguously to visual encodings \cite{ziemkiewiczEmbeddingInformationVisualization2009}, and provide a methodology for verifying that changes in data should correspond to changes in the visualization.


\subsection{Structure Preservation In Software}
\label{sec:related-work:software}

The visual algorithms implemented in building block libraries \footnote{a term introduced by Wongsuphasawat \cite{wongsuphasawatNavigatingWideWorld2021,wongsuphasawatNavigatingWideWorld2020} to describe libraries that provide modular components for building elements of a visualization} are generally assumed to preserve the underlying continuity of their input data, as described in taxonomies of visualization algorithms by Chi\cite{chiTaxonomyVisualizationTechniques2000} and by Troy and M\"{o}ller \cite{toryRethinkingVisualizationHighlevel2004}. 
In libraries such as Matplotlib\cite{hunterMatplotlib2DGraphics2007} and D3\cite{bostockD3DataDrivenDocuments2011}, assumptions about connectivity are embedded in the interfaces such that the API is inconsistent across plot types. For example in Matplotlib methods for updating data and parameters for controlling aesthetics differ between (1D) line based plotting methods and (0D) marker based methods. While libraries like VTK\cite{hanwellVisualizationToolkitVTK2015,geveciVTK2012} and Bokeh\cite{bokehdevelopmentteamBokehPythonLibrary2020} provide a way to express the connectivity of the data, and therefore can embed that information in their visual algorithms, their charting APIs are similar to the continuity dependent APIs of other building block libraries. 
In contrast, domain specific libraries are designed with the assumption of continuities that are common in the domain \cite{heerSoftwareDesignPatterns2006}, and therefore can somewhat restrict their API to choices that are appropriate for that domain. For example, a tabular structure of discrete rows, as illustrated in \autoref{fig:connectivity:ktypes}, is assumed by Tableau\cite{stoltePolarisSystemQuery2002, hanrahanVizQLLanguageQuery2006, mackinlayShowMeAutomatic2007} and excel charts, which try to suggest structure preserving visualizations, and ggplot\cite{wickhamGgplot2ElegantGraphics2016} and vega\cite{satyanarayanDeclarativeInteractionDesign2014}, which leave the choice to the user. Image libraries such as Napari\cite{sofroniewNapariNapari045rc12021} and ImageJ\cite{schneiderNIHImageImageJ2012} and the ImagePlot\cite{studiesCulturevisImageplot2021} art plugin assume that the input is a 2D continuous image. Networking libraries such as gephi\cite{bastianGephiOpenSource2009} and networkx\cite{hagbergExploringNetworkStructure2008} assume a graph-like structure. By assuming the structure of their data, these domain specific libraries can provide interfaces that are easier to navigate due to more similar specification notation \cite{kruchtenMetricsBasedEvaluationComparison2024} because the visual algorithms can share assumptions about the underlying data. We propose that the cohesiveness of domain specific library APIs is obtainable via a uniform data model, while the expressivity\cite{mackinlayAutomatingDesignGraphical1986} of building block libraries can be preserved by using category theory specify functional components \cite{huHowFunctionalProgramming2015, hughesWhyFunctionalProgramming1989} that are easily composable due to consistent software interfaces \cite{wielsManagementEvolvingSpecifications1998,yorgeyMonoidsThemeVariations2012}.

\section{Uniform Abstraction for Data \& Graphics}
\label{sec:atct}
Generally, preserving structure means that a visualization is expected to preserve the field properties and connectivity of the corresponding dataset. We use \textcolor{fiber}{\textbf{field}}\footnote{In this paper, color in definitions, equations, and visualizations is used to group conceptually related terms\cite{headMathAugmentationHow2022}, e.g. \textcolor{fiber}{field} and \textcolor{fiber}{fiber}.} to denote the set of values of the same type, e.g. date, location, stations, and precipitation in a weather station dataset and \textcolor{base}{\textbf{connectivity}} to denote how elements are arranged in a dataset\cite{wilkinsonGrammarGraphics2005}, e.g. as 0D ($\bullet$) points, along a 1D (--) line, in a 2D ($\blacksquare$) surface, as a 3D trajectory, a volumetric cube, or as a network. We propose a mathematical abstraction of the data input and pre-rendered graphic output that provides a uniform highly generalizable interface for describing topology and fields; expresses how to verify that data connectivity is preserved on subset, distributed, and streaming data representations; and formalizes the expectation of a correspondence between data and visual elements. Using these abstractions allows us to express the structure of data and graphics using the type signature $\textcolor{section}{\texttt{dataset}}: \textcolor{base}{\texttt{connectivity}} \rightarrow \textcolor{fiber}{\texttt{fields}}$, which means visualization algorithms can check the data input against a signature to ensure that the assumptions of the data match the assumptions of the algorithm. This typing system also extends to pre-rendered graphic output, allowing us to develop the structure preservation framework that ensures that output fields are equivalent to the input fields and that the connectivity of the output graphic matches the connectivity of the input.

\subsection{Abstract Data Representation}
\label{sec:atct:fiber-bundles}
We model data using a mathematical representation of data that can encode topological properties, field types, and data values in a uniform manner using a structure from algebraic topology called a fiber bundle.  We extend Butler's proposal of using bundles as an abstraction for visualization data\cite{butlerVectorBundleClassesForm1992,butlerVisualizationModelBased1989} by incorporating Spivak's methodology for encoding named data types from his fiber bundle representation of relational databases \cite{spivakDatabasesAreCategories2010,spivakSimplicialDatabases2009}. 

Encoding the data as fiber bundle provides a uniform interface for individually specifying the connectivity and field properties and the relationship between these spaces $\textcolor{section}{\texttt{dataset}}: \textcolor{base}{\texttt{connectivity}} \textcolor{section}{\rightarrow} \textcolor{fiber}{\texttt{field}}$ for datasets with arbitrary structures. This is because a \textbf{fiber bundle} $\dfiberc \textcolor{total}{\hookrightarrow} \dtotalc \textcolor{total}{\overset{\pi}\twoheadrightarrow}\dbasec$ is a structure with topological spaces $\dtotalc, \dfiberc, \dbasec$ and bundle projection map $\pi: \dtotalc \rightarrow \dbasec$ \cite{spanierAlgebraicTopology1989}(Supplement~\autoref{supp-sec:supplement:fiber_bundles}) and we propose that the \textcolor{total}{total space} \dtotalc\ of a bundle can encode the mathematical space in which a dataset is embedded, the \textcolor{base}{base space} \dbasec\ can encode the connectivity of the dataset, the \textcolor{fiber}{fiber space} \dfiberc\ can encode the field properties of the dataset, and that the datasets can be encoded as \textcolor{section}{section functions} \dsectionc\ from the base space to the fiber space. 

Fiber bundle spaces can also be partitioned into sub bundles, called local trivializations, with transition maps to glue the local trivializations back together. For example, we can unwrap a sphere into a plane to show that a continuous map exists between a segment of the globe and a corresponding choropleth, thereby addressing Wilkinson's concern that connectivity cannot be tested in a generalizable way because continuous maps do not always exist, specifically between a planar geographic maps and the globe \cite{wilkinsonGrammarGraphics2005}. The subsets of the fiber bundle and the transition maps between these subsets are encoded in an atlas\cite{ghristElementaryAppliedTopology2014} and the notion of an atlas can be incorporated into the data container, as discussed in \autoref{sec:atct:sheaves}.

\subsubsection{\textcolor{base}{Connectivity: Base Space \dbase}}
\label{sec:atct:fb:base}
\begin{figure}[h]
  \includegraphics[width=1\columnwidth]{k_different_types.pdf}
  \caption{This weather station data has multiple embedded continuities - points at each time and position, timeseries at each position, and maps at each time. The corresponding visualizations - bar chart, timeseries, and map - each preserve the continuity of the subset of the data they visualize by not introducing or leaving out values and preserving the relative positioning of continuous values.}
  \label{fig:connectivity:ktypes}
\end{figure}


The base space \dbase\ can be thought of as an
indexing set for the data -- where tabular data may have a discrete indexing set, time-series are indexed over one-dimensional time, and spatial fields are indexed over $d$-dimensional space, as illustrated in \autoref{fig:connectivity:ktypes} -- because the index is encoding the connectivity. Framing this in the language of topology provides us with an abstraction of our data that can be unified over data representation (in memory, distributed, streaming) and over different connectivity types (discrete, $d$-dimensional). There are many formal definitions of a topology; we choose to work with the neighborhood space axiomatization (Supplement~\autoref{supp-sec:topology}) because it is most analogous to the data access model of index (element) in subset (neighborhood) of all indices (base space).

\begin{definition}
We define the \textcolor{base}{base space} to be an indexing space \dbasec\ with indices $\dbasepoint$ and a neighborhood topology. A neighborhood of  $\dbasepoint$ has to contain $\dbasepoint$, can grow, or shrink, and every neighborhood contains smaller neighborhoods of points adjacent to $\dbasepoint$. \end{definition}


In the indexing cube in \autoref{fig:connectivity:ktypes}, the brown surface and blue rectangle cube are both neighborhoods of the index for the measurement in Albany on June 09. The blue rectangle is also a neighborhood of the index for the measurement in Albany on March 09. The indexing cube is a neighborhood for both of these indices. Using Munzner's key-value based semantic indexing\cite{munznerWhatDataAbstraction2014}, the data in \autoref{fig:connectivity:ktypes} could be subset into sets of timeseries where the key would be $station$, or subset into maps where the key would be $date$, or subset into station records where the keys are $(date, latitude, longitude)$. Using a connectivity model expresses how different labeling schemes can refer to the same indicies, e.g. how 0-360 and 180E-180W are two ways of labeling longitude or how (date, lat, lon) and (date, station) refer to the same data. In this paper we usually model the indexing space as CW-complexes, which are a class of topological spaces built by gluing together n-dimensional balls (which include points, intervals, filled circles, filled spheres, etc.) using continuous attaching maps. The topology can be defined to have a resolution equal to the number of indices in a dataset\cite{munkresElementsAlgebraicTopology1984} such that the key (continuity)-value (data) pairing is always preserved. 


\paragraph{Base Space Data Type}
We encapsulate the desired properties of the connectivity as the category $\mathcal{\dbase}$ (Supplement~\autoref{supp-sec:category}) to express that these are the properties that are required as part of the implementation of the base space data type. The standard construction to create a category from a topological space takes the objects to be all the neighborhoods, and morphisms to be the inclusion maps\cite{riehlCategoryTheoryContext2016}. For neighborhoods $\openset_1, \openset_2$, there will be at most one morphism $\dfunchc \in Hom(\opensetc_1, \opensetc_2)$. We denote the objects as $\dbasec$ when the underlying fiber bundle is trivial (Supplement~\autoref{supp-sec:appendix:bundle_triviality}), which is the case for the bundles discussed in this paper.
\subsubsection{\textcolor{fiber}{Field Data Types: Fiber Space \dfiber}}
\label{sec:atct:fb:fiber}
Spivak shows that data typing can be expressed in a categorical framework in his fiber bundle formulation of tables in relational databases \cite{spivakDatabasesAreCategories2010,spivakSimplicialDatabases2009}. We adopt Spivak's definitions of \textit{type}, \textit{schema}, \textit{record}, and \textit{table} because that allows us to use a dimension agnostic named typing system for the dataset fields that is consistent with the abstraction we are using to express the continuity. Broadly, Spivak defines \textit{types} as map $\sigma$ from type name (e.g. $temperature \in C$) to data type ($\texttt{float} \in \texttt{DT}$) and maps $\pi_{\sigma}, \pi$ from type domain ($\mathbb{R}\subseteq \mathscr{U_{\sigma}} \subseteq \mathscr{U}$) to name and type, \textit{schemas} as maps from type name to data type ($temperature: \texttt{float}$), \textit{records} as maps from name to value ($temperature:152.0^{\circ} F$) and \textit{tables} as maps from index to record \cite{spivakSimplicialDatabases2009} (see Supplement~\autoref{supp-sec:spivak} for details):
\begin{equation*}
    \begin{tabular}{|c|c|c|}
      \hline
       types and schema & \textcolor{fiber}{record} & \textcolor{section}{table}\\
      \hline
      \begin{tikzcd}[ampersand replacement=\&, sep=huge]
        \textcolor{total}{\mathcal{U}_\sigma} \arrow[d, "\pi_{\sigma}"', color=total, two heads] \arrow[r, dotted] \arrow[rd, dotted] \& \textcolor{total}{\mathcal{U}} \arrow[d, "\pi"', color=total, two heads] \\
        \textcolor{base}{C} \arrow[r, "\sigma"']                                                                                   \& \textcolor{base}{\texttt{DT}}
      \end{tikzcd}
    &
    \begin{tikzcd}[ampersand replacement=\&, sep=huge]
      \textcolor{total}{\mathcal{U}_{\sigma}} \arrow[d, "\pi_{\sigma}"', color=total, two heads] \\
      \textcolor{base}{C} \arrow[u, "\delement"', dashed, bend right, color=fiber]
      \end{tikzcd}
    &
    \begin{tikzcd}[ampersand replacement=\&, sep=huge]
      \dfiberc\coloneqq \textcolor{total}{\Gamma^{\pi}(\sigma)} \arrow[d, "\pi"', color=total, two heads] \\
      \dbasec \arrow[u, "\dsectionc"', dashed, bend right, color=section]
      \end{tikzcd}\\
      \hline
    \end{tabular}
\end{equation*}

\begin{definition}
  \label{def:fiber}
  We define the \textcolor{fiber}{fiber space} \dfiberc\ as the space of all possible data records $ \dfiberc \coloneqq \{\delement: \fnames \rightarrow \mathscr{U}_{\sigma} \bigm{\vert} \pi_{\sigma}(\delement(\fname)) = \fname\;for\;all\; \fname \in \fnames \}$ such that the preimage of a point is the corresponding data type domain $\pi^{-1}(\dbasepoint) = \dfiber_{k} = \mathscr{U}_{{\sigma}_{\dbasepoint}}$. 
\end{definition}
Adopting Spivak's fiber bundle construction of types allows our model to reuse types so long as the field names are distinct and that field values can be accessed by field name,  since those are sections on $\mathcal{U}_{\sigma}$. Furthermore, since domains $\mathscr{U}_{{\sigma}}$ of types are a mathematical space, multi-dimensional fields can be encoded in the same manner as single dimensional fields and fields can have different names but the same type.

\paragraph{Fiber Space Data Type}
As with the base space category $\mathcal{\dbase}$, we propose a fiber category $\mathcal{\dfiberc}$ to encapsulate the properties of the field types of the data. The fiber category has a single object $\dfiberc$ of an arbitrary data type, morphisms on the fiber object $\dfunctc \in Hom(\dfiberc, \dfiberc)$.  

\subsubsection{Composition: Adding and Expanding Records}
\label{sec:atct:composition}
A common use of category theory in software engineering is the specification of modular components \cite{wielsManagementEvolvingSpecifications1998} to build systems where the structure preserved by components is preserved in the composition of the components. Index spaces $\dbase_a$ and $\dbase_b$ can be composed by taking the fibered coproduct $\oplus: \dbase_{a}, \dbase_{b} \mapsto \dbase_{a} \sqcup_{\dbase_{c}} \dbase_{b}$ over a shared space $\dbase_{c}$. Fields $\dfiber_a$ and $\dfiber_b$ can be composed by taking the fiber product $\otimes: \dfiber_{a}, \dfiber_{b} \mapsto \dfiber_{a} \times_{\dfiber_c} \dfiber_{b}$ over a shared fiber $\dfiber_c$. 
\begin{figure}[H]
  \includegraphics*[width=1\columnwidth]{figures/tex/more_records.pdf}
  \caption{Records are added to the dataset as the \textit{fibered coproduct} $\dbase_{a}\sqcup_{\dbase_c} \dbase_{b}$ of index spaces. Fields are added to each record as the \textit{fiber product} $\dfiber_{a}\times_{\dfiber_c} \dfiber_{b}$ of fiber spaces.}
  \label{fig:atct:joint}
\end{figure}
The fibered coproduct is a generalization of inserting non-duplicated records into a database, while the fiber product is a generalization of a database join on a key. For example, in \autoref{fig:atct:joint}, the neighborhood $\dbase_c$ is present in both $\dbase_a$ and $\dbase_b$; therefore $\dbase_a$ and $\dbase_b$ are glued together where they overlap at $\dbase_c$.  Two spaces have been combined correctly if every point in the overlap  $\dbasepoint \in \dbase_{c}$ is present in each of the spaces $\dbasepoint \in \dbase_{a}$ and $\dbasepoint \in \dbase_{b}$ such that it is present in the joined space $\dbasepoint \in (\dbase_{a}\sqcup
_{\dbase_{c}} \dbase_{b})$. In \autoref{fig:atct:joint}, the (temperature, time, pressure) record separates into (temperature, time) and (pressure, time) records that share the same time. Furthermore this time is the same whether it is obtained from the (temperature, time) or (pressure, time) record. Generally, the test for joining fields is that when a record is present in a fiber $\delement_{c} \in \pi_c(\dfiber_{a}), \delement_c \in \pi_c(\dfiber_{b})$ then it is in the joint record $\delement_{c} \in \pi_c(\dfiber_{a}\times_{\dfiber_c}\dfiber_{b})$ when is in the field $\dfiber_c$ being joined on. Verifying that indices are joined correctly allows us to reliably build larger datasets out of smaller ones, such as in the case of distributed and on demand datasets. Verifying that fields are joined together correctly for the same record allows us to reliably combine multiple datasets together on shared properties, e.g. growing the weather station data in \autoref{fig:connectivity:ktypes} from a temporal to spatial dataset by adding the weather at each location at each time.

\subsubsection{\textcolor{section}{Data: Section $\dsection$}}
\label{sec:atct:fb:sections}
We encode data as a \textcolor{section}{section} \dsectionc\ of a bundle because this allows us to incorporate the topology and field types in the data definition.  For a neighborhood $\openset$ in the base space $\dbase$, a section is a function $\dsection:\openset\to\dfiber$. The set of all sections over a specific neighborhood is conventionally denoted as $\cgamma{\opensetc}{\dtotalc} \coloneqq \textcolor{set}{\big\{\textcolor{section}{\dsectionc : \opensetc \rightarrow \dtotalc\textcolor{black}{\vert_{\opensetc}}} \; \bigm{\vert} \textcolor{total}{\pi(\dsectionc(\dbasepointc))} = \dbasepointc \text{ for all } \dbasepointc \in \opensetc \big\}}$. 

\begin{definition} We define the \textcolor{section}{section} \dsectionc\ as the map $\dsectionc: \dbasepointc \mapsto \delementc$ from each point $\dbasepoint_{i} \in \openset$ to a record in the fiber space $\delement \in \dfiber_{\dbasepoint_{i}}$ over $\dbasepoint_{i}$. 
\end{definition} 

We can model data as structures that map from an index like point $\dbasepoint$ to a data record $\delement$, and encapsulate multiple datasets with the same fiber and base space as different sections of the same bundle. When data exists for every point in the base space (the section is continuous), we can define a global section $\dsection: \dbase \rightarrow \dfiber \in \Gamma_{\dbase}(\dtotal)$ which we translate into a data signature of the form $\textcolor{section}{\texttt{dataset}}: \textcolor{base}{\texttt{connectivity}} \textcolor{section}{\rightarrow} \textcolor{fiber}{\texttt{field}}$ where $\dsectionc=\textcolor{section}{\texttt{dataset}}$, $\dbasec=\textcolor{base}{\texttt{connectivity}}$ and $\dfiberc=\textcolor{fiber}{\texttt{fields}}$. When the section is not global, we can use the fiber bundle property of local-triviality to define local sections $\dsection\vert_{\openset} \in \Gamma_{\openset_{\dbasepoint}}(\dtotal)$ over a point. Most data sets can be encoded as a collection of local sections $\{\dsection\vert_{\openset_{\dbasepoint}} \text{for all } \dbasepoint \in \dbase \}$ where each section has the signature $\texttt{dataset:}{\texttt{connectivity}} \rightarrow {\texttt{field}}$.

\subsubsection{Example: Uniform Abstract Graphic Representation}
One of the reasons we use fiber bundles as an abstraction is that they are general enough that we can also encode the output of a visual algorithm as a bundle. We denote the output as a graphic, but the use of bundles allows us to generalize to output on any display space, such as a screen or 3D print. The graphic bundle $\gfiberc \textcolor{total}{\hookrightarrow} \gtotalc \textcolor{total}{\overset{\pi}\twoheadrightarrow}\gbasec$ consists of the total space \gtotalc\ which is an abstraction of an ideal (infinite resolution) space into which the graphic can be rendered, the base space \gbasec\ which is a parameterization of the display area, e.g. the inked bounding box in the cairo \cite{CairographicsOrg} rendering library, and the fiber space \gfiberc\ which is an abstraction of the renderer fields (e.g. a 2 dimension screen has pixels that can be parameterized $ \gfiber=\{x,\,y\,z,\,r,\,g,\,b,\,a\}$). As with data, we model the graphic specifications as sections $\gsection$ of the graphic bundle $\cgamma{\opensetgc}{\gtotalc} \coloneqq \textcolor{set}{\big\{\textcolor{section}{\gsectionc : \opensetgc \rightarrow \gtotalc\textcolor{black}{\vert_{\opensetgc}}} \; \bigm{\vert} \textcolor{total}{\pi(\gsectionc(\gbasepointc))} = \gbasepointc \text{ for all } \gbasepointc \in \opensetgc \big\}}$. The sections $\gsection$ map from a point in an openset in the graphic space $\gbasepoint \in \opensetg \subseteq \gbase$ to a point in the graphic fiber $\gfiber$. The section evaluated on a single point $\gbasepoint$ returns a single graphic record, e.g., one pixel in an ideal resolution space. In our model, the unevaluated graphic section is passed from the visualization library component to the renderer to generate graphics.

  \caption{The pixels in the scatter marker are the records returned by the section function $\gsection$ over the points $\gbasepoint \in \gbase$.\label{fig:atct:fb:graphic}}
\end{figure}

In \autoref{fig:atct:fb:graphic}, the bundle $\gtotal$ is a trivial bundle so $\gtotal=\gfiber \times \gbase$; therefore, the section function $\gsection:\gbase \rightarrow \gfiber$ maps into the fiber for a simplified 2D RGB infinite resolution pre-render space and returns the $\{x,y,r,g,b\}$ values of a pixel in an infinite resolution space. In \autoref{fig:atct:fb:graphic} these pixels are approximated as the small colored boxes. Each pixel is the output of the $\gsection(\gbasepoint)$ section that intersects the box. The set of all pixels returned by a section evaluated on a given visual base space $\gsection|_{\gbase}$ can yield a visual element, such as a marker, line, or piece of a glyph and in $\autoref{fig:atct:fb:graphic}$ is a blue circle with a black edge. While \autoref{fig:atct:fb:graphic} illustrates a highly idealized space with no overlaps, overlaps can be managed via a fiber element $\gfiber_{z}$ for ordering. It is left to the renderer to choose how to blend $\gfiber_{z}$ and $\gfiber_{a}$ layers.


\subsubsection{Data transformations}
\label{sec:atct:transforms}
Data is often codified as the set of values and the mathematical actions (Supplement~\autoref{supp-sec:actions}) on those values, e.g., the Steven's measurement scales define nominal, ordinal, interval, and ratio data by the allowed operations on the data \cite{stevensTheoryScalesMeasurement1946}; other researchers have since expanded the scales to encapsulate more types of structure \cite{leaFormalizationMeasurementScale1971, thomasMathematizationNotMeasurement2014}. We separate data transformations into transformations on the base space $(\dfunchc, \dfuncpullc)$ and transformations on the fiber space $\dfunctc$.


\paragraph{\textcolor{base}{\textbf{Connectivity}}}
\noindent
The base space transformation $\dfunchc$ is a continuous map between two base spaces: $\dfunchc: \dbasec' \to \dbasec$. For any such map between topological spaces, a bundle over $\dbase$ transforms into the pullback bundle over $\dbase'$. A section $\dsection:\dbase\to\dtotal$ over $\dbase$ is transformed into a section $\dsection'=\dsection\circ \dfunch$ through the pullback \dfuncpull. This means that the index of a record changes $\dbasepointc=\dfunchc(\dbasepointc')$ but the values in the record are unmodified $\dsectionc(\dbasepointc) = \dfuncpullc\dsectionc(\dbasepointc^{\prime}) = \dsectionc(\dfunchc(\dbasepointc^{\prime}))$.

\paragraph{\textcolor{fiber}{\textbf{Records}}}
The fiber space transformation $\dfunctc$ is a continuous map between two base spaces: $\dfunctc: \dfiberc \to \dfiberc'$. The function $\dfunct$ transforms $\dsection(\dbasepoint) \in \dfiber$ into $\dfunct(\dsection(\dbasepoint))\in \dfiber'$. To be structure preserving, $\dfunct$ must be a morphism in the category containing the fiber. 

\begin{table}[H]
  \begin{tabular}{|lll|}\hline
      scale & operators & sample constraint \\ \hline
      nominal & $=,\neq$ &  $\dsectionc(\dbasepointc_1) \neq \dsectionc(\dbasepointc_2)\implies \dfunctc (\dsectionc(\dbasepointc_1)) \neq\dfunctc(\dsectionc(\dbasepointc_2))$\\
      ordinal & $<, \leq, \geq, >$ &  $\dsectionc(\dbasepointc_1) \leq \dsectionc(\dbasepointc_2) \implies \dfunctc (\dsectionc(\dbasepointc_1)) \leq \dfunctc(\dsectionc(\dbasepointc_2)$) \\
      interval & $+, -$ &  $\dfunctc(\dsectionc(\dbasepointc) + C) = \dfunctc(\dsectionc(\dbasepointc)) + C$ \\
      ratio & $*,/$ &  $\dfunctc(\dsectionc(\dbasepointc) * C) = \dfunctc(\dsectionc(\dbasepointc))*C $\\ \hline
  \end{tabular}
  \caption{To be structure preserving with respect to a given measurement scale, $\dfunct$ must commute with the operators of that scale.}
  \label{tab:appendix:summary:stevens}
\end{table}

\paragraph{\textcolor{section}{\textbf{Connectivity and Records}}}
\noindent 

We define a full data transformation as one that induces both a remapping of the index space and a change in the data values: $\dfuncc: \dsectionc\mapsto\dfunctc\circ\dsectionc\circ\dfunchc$. Fiber transformations applied to different factors of the fiber space (as a product data type) compose into simultaneous transformations of the composite fiber, while base space transformations applied to different components of the base space compose into simultaneous transformations of the composite base space. We can define a composite transform where each fiber transform in a collection can be applied to a different fiber field. These function compositions are implemented in the same manner as record compositions, as described in \autoref{sec:atct:composition}. 

\begin{figure}[!h]
  \includegraphics[width=1\columnwidth]{actions.pdf}
  \caption{The actions on the base and fiber space commute such that the data can be permuted then scaled or scaled then permuted.} \label{fig:actions}
\end{figure}

The component actions are independent such that they can be applied in either order. In \autoref{fig:actions}, the action $\dfunch, \dfuncpull$ permutes the indexing space such that locations of the bars relative to each other change. The action $\dfunct$ doubles the data value associated with the height of the bar. Because these are both structure preserving actions, the order in which the bars are transformed - permute then scale or scale then permute - does not change the final image. 

\subsection{Abstract Data Containers}
\label{sec:atct:sheaves}

While bundles provide a way to describe the structure of the data, sheaves are a mathematical way of describing the data container. Sheaves are an algebraic data structure that provides a way of abstractly discussing the bookkeeping that data containers must implement to keep track of the continuity of the data \cite{ghristElementaryAppliedTopology2014}. This is because sheaves and presheaves are functors  (Supplement~\autoref{supp-sec:functors}), which are mappings between categories that preserve the domains, codomains, composition, and identities of the morphisms within the category\cite{riehlCategoryTheoryContext2016}. Using sheaves as the data container abstraction facilitates representational invariance, as formulated by Kindlemann and Scheidegger\cite{kindlmannAlgebraicProcessVisualization2014}, since the container level is uniformly specified as satisfying presheaf and sheaf constraints (Supplement~\autoref{supp-sec:preasheaf_sheave}). When a data container satisfies these constraints, the subsets and whole data space have the same mathematical properties, e.g. morphisms, such that the framework in this paper applies whether the data is in memory, distributed, streaming, or on-demand.

Codifying the data container as the functor $\sheafc_{\dbasec, \dtotalc}: \opensetc \textcolor{sheaf}{\rightarrow} \cgamma{\opensetc}{\dtotalc}$ expresses that the container is a structure preserving map between the objects in the index space and sets of data records and that the container preserves the maps between index space objects and the maps between data records. The data container has the characteristics of a presheaf, which is a functor from an arbitrary category to the set category. A property of presheaves is that they are contravariant, meaning that the morphisms between the input objects $\dfunch$ go in the opposite direction from the morphisms between the output objects $\dfuncpull$, as illustrated in \autoref{fig:actions}. As discussed in \autoref{sec:atct:transforms}, $\dfunch$ changes the index and $\dfuncpull$ aligns the records by "pulling back" records to where they belong in the new index. The map between sets is contravariant to the map over indices because not all sets exist over all indices, e.g. when $\dfunch$ is an inclusion mapping that returns a subset of the full index. Sheaves are presheaves that satisfy the gluing and locality axioms. The gluing axiom says that a distributed representation of a dataset, which is a set of local sections, is equivalent to a section over the union of the opensets of the local sections. The gluing axiom can also be used to generate the gluing rules used to construct non-trivial bundles from the set of trivial local sections. The locality axiom asserts that the glued section function is equivalent to a function over the union if they evaluate to the same values. When satisfied, these axioms ensure that a data container is managing distributed and streaming data in a structure preserving way.
The advantage of using these constraints for specifying structure preserving data constraints is that the test for whether a container implements these constraints is trivial:

\section{Connectivity Preserving Map $\vindexc$}
\label{sec:atct:xi}
From the early maps of stars on, there has been an expectation that the connectivity in visualizations matches the connectivity of the underlying data\cite{friendlyBriefHistoryData2008}. This is somewhat codified in the notion of readability\cite{ziemkiewiczEmbeddingInformationVisualization2009}, which is that visual elements must correspond to distinct data elements.  We propose that if connectivity is preserved then there should exist a continuous map \vindexc\ that expresses the correspondence between graphic and data indexing spaces. 

\begin{definition}
\label{def:atct:xi}
We define the mapping \vindexc\ to be a continuous map $\vindexc: \opensetgc \textcolor{functor}{\rightarrow} \opensetc$ between a graphic neighborhood $\opensetgc \subseteq \gbasec$ and data neighborhood $\opensetc \subseteq \dbasec$. 
\end{definition}

\noindent The set of points in graphic space that correspond to each point in data space is $\vindexprec(\dbasepointc) = \textcolor{set}{\{}\gbasepointc\;\textcolor{set}{|}\; \vindexc(\gbasepointc) = \dbasepointc \textcolor{set}{\text{ for all }}\dbasepointc \in \dbase, \gbasepointc \in \gbasec\textcolor{set}{\}}$. We construct the map as going from graphic to data because that encodes the notion that every visual element traces back to the data in some way, but there may be more data available than what is shown in the graphic.

\begin{figure}[!h]
  \includegraphics[width=1\columnwidth]{xi_zoom.pdf}
  \caption{The function $\xi$ maps each point in each orange band on each indexing space $\gbase_0, \gbase_1, \gbase_2$ to the same point $\dbasepoint_i$ in the circle $\dbase$.\label{fig:artist:xi}}
\end{figure}
Each orange band in \autoref{fig:artist:xi} is defined as $\frac{7\pi}{4}\times [0,1]$ such that each point on each band maps to the same point $\vindex|_{\frac{7\pi}{4}\times [0,1]} = \frac{7\pi}{2}$. Despite discarding the cyclical information embedded in \dbase\, the spaces $\gbase_1$ and $\gbase_2$ have compatible structure to their images under \vindex\, as illustrated by the orange bands pulling back to a point.

Since we have defined a continuous function \vindex\ between two spaces $\dbase, \gbase$, we can construct functors that transport sheaves over each space to the other space\cite{harderLecturesAlgebraicGeometry2008}. This allows us to find what data we expect is being visualized at each graphic index location and what the graphic specification is for the data at each data index location.

\section{Artist: Structure Preserving Maps}
\label{sec:artist}
We build off of Kindlmann and Scheidegger's notion of equivariant\cite{kindlmannAlgebraicProcessVisualization2014} and Mackinlay's notion of homomorphic\cite{mackinlayAutomatingDesignGraphical1986} visualizations in our definition of structure preservation. 

\begin{definition}
We define a mapping from data to graphic to be \textbf{structure preserving} when connectivity is preserved, the structure on the data and the graphics is compatible, and changes to data and graphic commute.
\end{definition}
\noindent By mapping from data to graphic we mean, formally, a sheaf morphism from the sheaf of local sections on the data fiber bundle to the sheaf of local sections on the graphic fiber bundle: $f:\Gamma_{\dbase}(\dtotal)\to\Gamma_{\gbase}(\gtotal)$. We call the structure preserving mappings \textcolor{artist}{Artists}\footnote{Artists are named for the Artist class in Matplotlib\cite{hunterArchitectureOpenSource} because the Artist is the parent class of all the objects that assemble and manage visual elements such as points, lines, and shapes.}:
\begin{equation*}
  \label{eq:artist:hom_transport}
  \vartistc: \cgamma{\dbasec}{\dtotalc}\textcolor{artist}{\rightarrow} \imartist{\gbasec}{\gtotalc} \subset \cgamma{\gbasec}{\gtotalc}
\end{equation*}

\noindent The output of an artist \vartist\ is a restricted subset of graphic sections $\imartist{\gbasec}{\gtotalc}$ that are, by definition, only reachable through a structure preserving artist:$
  \imartist{\gbasec}{\gtotalc} \coloneqq
  \textcolor{set}{\{\gsectionc \mid\;\exists\;\dsectionc \in \cgamma{\dbasec}{\dtotalc}\;s.t.\;
  \vartistc(\dsectionc) = \gsectionc \}}$. We define this subset because the space of all sections $\Gamma_{\gbase}(\gtotal)$ is every possible way of setting pixels on a graphic base space, including non structure preserving settings such as setting all the pixels the same color. 
  
  A structurally preserving Artist is a \textit{natural transform} (Supplement~\autoref{supp-sec:natural-transform}), which is a structure preserving map of functors\cite{milewskiCategoryTheoryProgrammers2019}. Through the application of the transport functors (\vindexpush, \vindexpull), artists can be constructed such that an artist over data space $\vartist_{\dbase}: \dsection \mapsto \vindexpush \gsection$, an artist over graphic space $\vartist_{\gbase}: \vindexpull \dsection \mapsto \gsection$, and an artist $\vartist: \dsection \mapsto \gsection$ are equivalent (Supplement~\autoref{supp-sec:artist-transform}); therefore, the artist signature can be formulated as:
 \begin{align*}
  \textcolor{artist}{\texttt{Artist}}:&(\textcolor{section}{\texttt{dataset}}: \textcolor{base}{\texttt{connectivity}}\textcolor{section}{\rightarrow} \textcolor{fiber}{\texttt{fields}}) \\
  \textcolor{artist}{\rightarrow} & (\textcolor{section}{\texttt{graphic}}: \textcolor{base}{\texttt{connectivity}}\textcolor{section}{\rightarrow} \textcolor{fiber}{\texttt{fields}})
 \end{align*}
where \texttt{connectivity} and \texttt{field} are meta classes of compatible types. Defining the artist as a \textit{natural transform} means that the Artist must preserve the constraints on the input sheaves and output sections, meaning that actions $\dfunc$ on data and graphic must be equivariant (\autoref{sec:artist:equiv}), and the Artist must respect the sheaf constraints of \textit{locality} and \textit{gluing} for the input and equivalent output. The natural transform property also gives developers the flexibility to connect components in data space with components in graphic space, e.g. using the output of a component that assigns line color in data space to as the input of a line building component that dynamically resamples what is on screen in graphic space.

\subsection{Equivariance}
\label{sec:artist:equiv}
\begin{figure}[!h]
    \includegraphics[width=.5\textwidth, alt={Diagram of a dirichelet distribution evaluated on a triangle, transposed, and binned and the corresponding visualization output by the artist function.}]{artist_equiv.pdf}
    \caption{This artist maps a numerical section of the Dirichlet distribution to a corresponding graphical representation. The artist preserves connectivity by not breaking presheaf implementations of rotation $(\dfunch, \dfuncpull)$ of the indexing space $\dbase, \gbase$ and field structure by implementing the data to color mapping such that binning $(\dfunct)$ the data and mapping the bins to colors is equivalent to binning the colors directly. The subscripts denote the space the action occurs in.}\label{fig:artist}
\end{figure}

An artist is equivariant when changes to the input and output commute such that for all points $\gbasepointc^{\prime}\in \gbasec^{\prime}$, starting at an arbitrary data point $\dsectionc(\dbasepointc)$ and transforming it into a different data point and then into a graphic $\vartistc(\dfunctc_{\dtotalc}(\dsectionc(\dfunchc_{\dtotalc}(\vindexc(\gbasepointc^{\prime}))))) = \dfunctc_{\gtotalc}(\vartistc(\dsectionc(\vindexc(\dfunchc_{\gtotalc}(\gbasepointc^{\prime})))))$ is equivalent to transforming the original data point into a graphic and then transforming the graphic into another graphic. The function $\dfunch_{\gtotal}$ induces a change in graphic generating function that matches the change in data. For example, \autoref{fig:artist} shows an artist function $\vartist$ that maps a Dirichlet distribution on a triangular base space to a corresponding graphic. The artist is equivariant with respect to connectivity because the base space rotation actions $\dfunch, \dfuncpull$ commute such that rotating the data and then mapping it to graphic or rotating the graphic yield the same Dirichlet distribution graphic where the values have moved to new locations but not changed. The artist is also equivariant with respect to field structure because the binning action $\dfunct$ commutes such that binning and then color mapping the data yields an equivalent graph to binning the colormap directly. 

The graphic transformation $\dfunch_{\gtotal}$ is difficult to define because by definition it acts on a single record, e.g. a pixel in an idealized 2D screen. Instead, we describe the changes on the graphic side as changes in \textcolor{monoid}{measurements} \measurec\ of visual components. A superset of the Bertin visual variables \cite{bertinIIPropertiesGraphic2011}, measurable components are attributes of the rendered graphic that can be quantified, such as the color, position, shape, texture, or rotation angle of the graphic. For example, a measurement of a scatter marker could be its color (e.g. red) or its x position (e.g. 5). The measurable elements can only be computed over the entire preimage because these aspects, such as thickness or marker shape, refer to the entire visual element.

\begin{equation*}
  \label{eq:artist:inout:diagram}
  \begin{tikzcd}[row sep=small]
    \cgamma{\dbasec}{\dtotalc}
    \arrow[rr, "\vartistc", color=artist]
    \arrow[d, "\equivc"', color=monoid] &  &
    \imartist{\gbasec}{\gtotalc}
    \arrow[d, "\text{render}"]
    \arrow[lld, "\extractmc"', color=monoid, dashed] \\
    {\textcolor{set}{Hom}(\dbasec, \measurec)}  &  & \text{visualization}
    \arrow[ll, "\text{measure}",]
    \end{tikzcd}
\end{equation*}

We define an output \textcolor{action}{verification function} $\extractm: (\gsection \circ \vindexpre) \mapsto (\dbase \xrightarrow{\extractm_{\gsection}} \measure)$ that takes as input the section evaluated on all the graphic space associated with a point $\gsection_{\vindexpre\vert_{\dbasepoint}}$ and returns the corresponding \textcolor{action}{measurable visual components} $\measurec_{\dbasepointc}$. The verification function is equivalent to measuring components of the rendered image $\extractm = \text{measure}\circ \text{render}$, which means an alternative way of implementing the function when $\gbase$ is not accessible is by decomposing the output into its measurable components. We also introduce a function $\equivb: \dsection \mapsto (\dbase \xrightarrow{\equivb_{\dsection}} \measure)$ that maps data to the measurement space directly such that $\equivb_{\dsection}(\dbasepoint)$ is the expected set of measurements $\measure_{\dbasepoint}$. The pair of \textcolor{monoid}{verification functions} (\equivc, \extractmc) can be used to test that the expected encoding $\equivb_{\dsection}$ of the data matches the actual encoding $\extractm_{\gsection}$: $
    \equivc(\dsectionc)(\dbasepointc) = \extractmc(\vartistc(\dsectionc))(\dbasepointc) = \extractmc(\gsectionc\circ\vindexprec)(\dbasepointc)=\measurec_{\dbasepointc}$

 As introduced in \autoref{sec:atct:transforms}, the base space transformation \dfunch\ is invariant because $\dsection\vert_{\openset} = \dsection\vert_{\dfunch(\openset^{\prime})}$. This means that, for all points in the data $\dbasepoint \in \dbase$, the measurement should not change if only the base space is transformed $\equivc(\dsectionc)(\dfunchc(\dbasepointc^{\prime})) = \extractmc(\vartistc(\dsectionc))(\dbasepointc)$. In contrast, a change in sections induces an equivalent change in measurements $\equivc(\dfunctc(\dsectionc))(\dbasepointc) = \dfunctc_{\measurec}(\extractmc(\vartistc(\dsectionc))(\dbasepointc))$. The change in measurements $\dfunct_{\measure}$ is defined by the developer as the symmetry between data and graphic that the artist is expected to preserve.
\subsection{Composing Artists}\label{sec:artist:operators}
\begin{figure}[!h]
  \includegraphics[width=1\linewidth]{composition.pdf}
  \caption{The scatter plot of stations on the rain map from \autoref{fig:connectivity:ktypes} can either be generated as a composition of scatter and map drawing functions or a singular scatter and map drawing function.}
  \label{fig:composition}
\end{figure}

As shown in \autoref{fig:composition}, many graphics are compositions of smaller graphics, here a scatter plot and choropleth. Artists are composable because they preserve data and graphic composition \autoref{sec:atct:composition} and so must preserve the property of the coproduct that $Hom(K_1 \sqcup K_2, F) \simeq Hom(K_1, F) \times Hom(K_2, F)$ and the property of the product that $Hom(K, F_1 \times F_2) = Hom(K, F_1) \times Hom(K, F_2)$ \cite{maclaneCategoriesWorkingMathematician1978}, which we generalize as $Hom(\coprod_i K_i, \prod_j F_j) = \coprod_i \prod_j Hom(K_i, F_j)$ when composing with section functions $\dsection \in Hom(\dbase,\dfiber)$ to output graphic sections $\gsection \in Hom(\gbase, \gfiber)$. We use Hom notation here because it allows us to clearly express how composition with the artist differs for each space $\coprod_{i} \prod_{j} \vartistc(\homc{\dbasec_i}{\dfiberc_j}) = \vartistc(\homc{\coprod_i \dbasec_i}{\prod_j \dfiberc_j}) \implies \coprod_{i} \prod_{j} \homc{\gbasec_i}{\gfiberc_j} = \homc{\coprod_i \gbasec_i}{\prod_j \gfiberc_j}$ such that the same graphic can be generated from a singular artists on the entire data set. 
When combining spaces $\oplus: \dbase_{a}, \dbase_{b}$, we can construct the graphics section as the piecewise function
\begin{equation*}
  \label{eq:artist:plus:output}
  \gsectionc(\gbasepointc) \coloneqq \begin{cases} \gsectionc^{a}(\gbasepointc) & \gbasepointc \in \vindexprec(\dbasec^{a}) \\
    \gsectionc^{b}(\gbasepointc) & \gbasepointc \in \vindexprec(\dbasec^{b})
  \end{cases}
\end{equation*}

When combining fiber spaces $\otimes: \dfiber_{a}, \dfiber_{b}$, we construct the graphics function as the set of sections $\gsectionc^{a, b} = \{\gsectionc^{a}(\gbasepointc), \gsectionc^{b}(\gbasepointc)\}, \gbasepointc \in \vindexprec(\dbasec)$ but in the single output case, such as a figure to a screen, the output fiber is identical $D_1 \times {D_2} = D$ such that we can default to the piecewise construction. 

We can verify that base space composition was implemented correctly by verifying that the artist and composition of artists return the same graphic on points in the intersection $\dbase^{c}$.  Given $\dbasepointc_{a} \in \dbasec_{c} \subset \dbasec_{a}$ and $\dbasepointc_{b} \in \dbasec_{c} \subset \dbasec_{b}$, if $\dbasepointc_{a} = \dbasepointc_{b}$ then $\vartistc_{ab}(\dsectionc^{ab}(\dbasepointc_{a})) = \vartistc_{a}(\dsectionc^{a}(\dbasepointc_{a})) = \vartistc_{b}(\dsectionc^{b}(\dbasepointc_{b}))$ for all $\dbasepointc_{a}, \dbasepointc_{b} \in \dbasec_{a}\bigsqcup\limits_{\dbasec_{c}} \dbasec_{b}$. This tests whether graphics line up when drawn in parts, e.g. algorithms for visualizing streaming data \cite{krstajicVisualizationStreamingData2013} and techniques that rely on shifting or resampling the indexing space, such as  panning and zooming \cite{NekrasovskiEvaluationPanZoom2006}. We can verify that fiber space composition was implemented correctly by checking that any shared fiber $\dfiber^{c}$ is encoded consistently $\extractmc(\vartistc_{ab}(\dsectionc^{ab}(\dbasepointc)))\vert_{\dfiberc^{c}} =
    \extractmc(\vartistc_{a}(\dsectionc^{a}(\dbasepointc_{a})))\vert_{\dfiberc^{c}} = \extractmc(\vartistc_{b}(\dsectionc^{b}(\dbasepointc_{b})))\vert_{\dfiberc^{c}}$. 
This expectation of using the same encoding for the same variable is a generalization of the concept of consistency checking of multiple view encodings discussed by Qu and Hullman \cite{quKeepingMultipleViews2018}. In \autoref{fig:composition}, this is a check that, e.g., the scatter marker drawn by $\vartist_{a}$ is identical to the scatter marker drawn by $\vartist_{abcd}$. 

\subsubsection{Animation and Interactivity}
\label{sec:artist:anim_and_interactive}
Animation and interaction can be thought of as either composition of base spaces (more frames), fiber spaces (additions to the graphic such as annotations, or cursors), or a combination therefore; therefore structure preserving animations and interactions fall out as a consequence of implementing structure preserving artists such that it is structure preserving for all frames $\{\vartist(\dsection(\dbasepoint)) \mid\dbasepoint \in \dbase \}$. Panning, zooming, and scrolling rely on sheaf constraints that ensure that continuity is maintained as the data is subsetted, selection and hover require the corresponding information provided by the pushforward \vindexpush\ and pullback \vindexpull\ functors, and brushing, linking, and annotation rely on the bookkeeping of $\vindex$ and effectively implemented composition to ensure that the pieces in graphic are in sync such that they all map back to the same subset of the data or data point. 

\section{Constructing Structure Preserving Components}
\label{sec:construction}
\begin{figure}[!h]
  \includegraphics[width=1\columnwidth]{construction_comp.pdf}
  \caption{The artist function that draws the green point on the circle is composed of two stages $\vartist = \vchannel \circ \vmark$. The encoder \vchannelc\ maps the data values to visual values and the compositor \vmark\ generates the graphic specification for producing the sections of the circle. Default encodings, such as color and mark type, are here treated as pullbacks of constant values in the data.}
  \label{fig:construction:q}
\end{figure}

Artists naturally factorize into a two-stage process, starting with an encoding stage $\vchannel$ followed by a compositing stage $\vmark$. In the \textcolor{artist}{encoding} stage $\vchannelc$, a data bundle is treated as separable fields and each field is mapped to a measurable visual variable. In the encoding stage, many of the expected visual mappings $\equivb$ can be implemented inside the library. Factoring out the encoding stage leaves the \textcolor{artist}{compositing} stage $\vmarkc$ responsible for faithfully translating those measurable visual components into a visual element.

\subsection{Indexing Space}

We can use the data base space as an indexer into the graphic base space if we implement the graphic base space as if there was a deformation retraction (Supplement~\autoref{supp-sec:deform-retract}) from the graphic space to the data space: 

\begin{equation*}
\underbrace{\dbasec\times[0,1]^{n}}_{\gbasec} \textcolor{functor}{\xmapsto{\hspace{1em}\vindexc\hspace{1em}}} \dbasec, 
{\scriptscriptstyle n =} \begin{cases} {\scriptscriptstyle dim(\gbase) - dim(\dbase)} 
  &{\scriptscriptstyle dim(\dbase)<dim(\gbase)}\\ 0 
  & {\scriptscriptstyle otherwise} \end{cases}
\end{equation*}

where n is a thickening of the graphic base space $\gbase$ to account for the dimensionality of the output space because otherwise the data dimensionality $\dbase$ may be too small for a graphic representation. For example, a line is 1D but is a 2D glyph on a screen; therefore the graphic space $\gbase$ is constructed by multiplying the base space $\dbase$ with an interval $[0,1]$.

\subsection{Measurable Visual Components}
\label{sec:construction:vtotal}
We encapsulate the space of measurable components reachable through the encoding stage $\vchannel$ as a visual fiber bundle \textcolor{total}{$\vfiberc \hookrightarrow \vtotalc \xrightarrow{\pi} \dbasec$}. The  restricted fiber space $\vfiber$ of the bundle acts as the specification of the internal library representation of the measurable visual components. The space of visual sections $\cgamma{\opensetc}{\vtotalc} \coloneqq \textcolor{set}{\big\{\textcolor{section}{\vsectionc : \opensetc \rightarrow \vtotalc\textcolor{black}{\vert_{\opensetc}}} \; \bigm{\vert} \textcolor{total}{\pi(\vsectionc(\dbasepointc))} = \dbasepointc \text{ for all } \dbasepointc \in \opensetc \big\}}$ return a visual encoding $\vsectionc(\dbasepointc)$ corresponding to data record $\dsectionc(\dbasepointc)$.  Since the data bundle $\dtotal$ and visual bundle $\vtotal$ have the same continuity $\pi(\dsection(\dbasepoint)) = \pi(\vsection(\dbasepoint))$, they are considered structurally equivalent such that $\dtotal=\vtotal$. The distinguishing characteristic of $\vtotal$ is that it is part of the construction of the artist and therefore a part of the visualization library implementation. We propose that reusing the fibers $\vfiber$ across components facilitates standardizing internal types across the library and that this standardization improves maintainability.


\subsection{Component Encoders}
\label{sec:construction:nu}

As introduced in \autoref{sec:artist:equiv}, there is a set $\equivb$ of functions that map between data and corresponding visual encodings. We propose that for visualization library components to be structure preserving, they must implement a constrained subset of these encoding functions $\Gamma_{\dbase}(\dtotal) \xrightarrow{\vchannel} \Gamma_{\dbase})\vtotal)  \subset \Gamma_{\dbase}(\dtotal) \xrightarrow{\equivb}{Hom}(\dbase, \measure)$ that preserve the categorical structure (operators and morphisms) of the fiber and the continuity of the data section. As mentioned in \autoref{sec:construction:vtotal}, the total visual space is restricted to the space of data types internal to the library $\vfiber \subset \measure$ and sections are subsets of homsets $\Gamma(\dbase, \vtotal) \subset Hom(\dbase, \measure)$ because sections must be continuous. The encoding functions $\vchannel$ are fiber wise transforms such that $\pi(\dtotal) = \pi(\vchannel(\dtotal))$. A consequence of this property is that $\vchannelc$ can be constructed as a point wise transformation $\vchannelc: \dfiberc_{\dbasepointc} \rightarrow \vfiberc_{\dbasepoint}$, which means that means that a point in a single data fiber $\delement \in \dfiber_{\dbasepoint}$ can be mapped into a corresponding point in a visual fiber $\vfiber_{\dbasepoint}$. This means that an encoding function $\vchannel$ can convert a single record independent of the whole dataset.

Since $\dtotal$ and $\vtotal$ are structurally identical, any $\vtotal$ can be redefined as $\dtotal$; therefore any collection of $\vchannel$ functions can be composed such that they are equivalent to a $\vchannel$ that directly converts the input to the output:
\begin{equation*}
  \label{eq:construction:nu:fabrication}
  \begin{tikzcd}[row sep=small]
    \dfiberc_{\dbasepointc}
    \arrow[rr, "\vchannelc", color=artist]
    \arrow[rrrr, "\vchannelc^{\prime\prime}", dashed, bend right, color=artist] &  &
    \vfiberc_{\dbasepointc}\coloneqq{\dfiberc_{\dbasepointc}^{\prime}}
    \arrow[rr, "\vchannelc^{\prime}", color=artist] &  &
    \vfiberc^{\prime}_{\dbasepointc}
  \end{tikzcd}
\end{equation*}
Because the encoder $\vchannel$ is a natural transform between data and visual sheaves, $\vchannel$ can be composed in the same manner as artists as described in \autoref{sec:artist:operators}.

\subsection{Graphic Compositor}
The compositor function $\vmarkc$ transforms the measurable components into properties of a visual element. The compositing function $\vmarkc: \cgamma{\dbasec}{\vtotalc} \rightarrow \cgamma{\gbasec}{\gtotalc}$ transforms the sections of visual elements \vsection\ into sections of graphics \gsection. The compositing function is a map from sheaves over $\dbase$ to sheaves over $\gbase$ because the graphic section must be evaluated on all points in the graphic space to generate the visual element corresponding to a data record at a single point $\vartist(\dsection(\dbasepoint)) = \gsection(\vindexpre(\dbasepoint))$. Because the compositor $\vmark$ is a natural transform between visual and graphic sheaves, $\vmark$ can be composed in the same manner as artists as described in \autoref{sec:artist:operators}.

Since encoder functions are infinitely composable, as described in \autoref{eq:construction:nu:fabrication}, a new compositor function $\vmark$ can be constructed by pre-composing $\vchannel$ functions with the existing $\vmark$:
\begin{equation*}
  \label{eq:construction:q:fabrication}
  \begin{tikzcd}[row sep=small]
      \cgamma{\dbasec}{\vtotalc}
      \arrow[rr, "\vchannel", color=artist]
      \arrow[rrrr, "\vmarkc^{\prime}", bend right, color=artist, dashed] &  & \cgamma{\dbasec}{\vtotalc^{\prime}}
      \arrow[rr, "\vmarkc", color=artist] &  & \cgamma{\gbasec}{\gtotalc}
      \end{tikzcd}
\end{equation*}
The composition in \autoref{eq:construction:q:fabrication} means that different measurable components can yield the same visual elements. As shown in \autoref{fig:construction:q}, a set of  $\vchannel$ functions individually convert the values in the data record to visual components. Then the $\vmark$ function combines these visual encodings to produce a graphic section \gsection. When this section is evaluated on the graphic space associated with the data $\gsection(\vindexpre(\dbasepoint))$, it produces green pixels at the intersection of the x and y positions listed in \vsection. The composition rule in \autoref{eq:construction:q:fabrication} means that developers can implement $\vmark$ as drawing a circle or can implement a $\vmark$ that draws arbitrary shapes, and then provide different $\vchannel$ adapters, such as one that specifies that the shape is a circle.

\subsection{Verification}
\label{sec:construction:verification}

The factorization of the artist into the encoding and composition stages provides a framework for testing the conversion from data to visual component separately from the assembly of components into a glyph. For example, encoders built in this framework could be designed to ensure that a transformation from a non-colorblind safe colormap into a colorblind safe colormap would ensure that the colormaps are equivalent such that the transformation is not introducing a change in structure. Similarly, the composition stage can be tested to ensure that parameters are applied consistently across visual elements and, as with encoding, more accessible elements can be swapped in without introducing a change in structure.

\subsubsection{Encoder}
\label{sec:construction:nu:verification}
\begin{figure}[!h]
  \includegraphics{encoder.pdf}
  \caption{The encoders $\vchannel_{color, marker}$ and $\vchannel_{color}$  correctly encode JFK INTL AP as steelblue if they return steelblue and it matches the reference color \#4682b4. An external testing library can follow a similar procedure using \equivb\ functions to test library encoders \vchannelc.}
  \label{fig:verify_encoder}
\end{figure}
An encoding function $\vchannelc$ is equivariant if the change in data, as defined in \autoref{sec:atct:transforms}, and change in visual components are equivariant. Since $\dtotal$ and $\vtotal$ are over the same base space and are point wise, the base space change $\dfunch_{\dtotal}$ applies to both sides of the equation $\vchannelc(\dsectionc_{\dtotalc}(\dfunchc_{\dbasec}(\dbasepointc^{\prime}))) = \vsectionc(\dfunchc_{\dbasec}(\dbasepointc^{\prime}))$ and therefore there should not be a change in encoding. On the other hand, a change in the data values $\dfunct_{\dtotal}$ must have an equivalent change in visual components $\dfunctc_{\vtotalc} \vchannelc(\dsectionc(\dbasepoint)) = \vchannelc(\dfunctc_{\dtotalc}(\dsectionc(\dbasepointc)))$ The change in visual components $\dfunct_{\vtotal}$ is dependent both on $\dfunct_{\dtotal}$ and the choice of visual encoding. 


\subsubsection{Compositor}
\label{sec:construction:q:verification}
\begin{figure}[!h]
    \center
  \includegraphics{compositor.pdf}
  \caption{The compositors $\vmark_{square}$ amd $\vmark_{circle}$ correctly compose the visual parameter "steelblue" in their outputs if the measured color is the same in the output of both compositors is equal to the reference color \#4682b4.}
  \label{fig:verify_compositor}
\end{figure}

An advantage of factoring out encoding and verification, as discussed in \autoref{sec:construction:nu:verification}, is that the responsibility of the compositor can be scoped to translating measurable components into visual elements. One way of verifying that a compositor is consistent is by verifying that it passes through one encoding even while changing others. For example, when $\vmark_{ab}=\vmark_{ac}$ then the output should differ in the same measurable components as $\vsectionc_{ab}$ and $\vsection_{ac}$.

A compositor function \vmark\ is equivariant if the renderer output changes in a way equivariant to the data transformation. This means that a change in base space $\dfunch_{\dtotalc}$ should have an equivalent change in visual element base space, which means that there should be no change in visual measurement $\vsectionc(\dfunchc_{\dbasec}(\dbasepointc^{\prime})) = \extractmc(\vmarkc(\vsectionc)(\dfunch_{\dbasec}(\vindexprec^{\dbasepointc}))) = \measurec_{\dbasepointc}$. As discussed in \autoref{sec:artist:equiv}, the change in base space may induce a change in locations of measurements relative to each other in the output; this can be verified via checking that all the measurements have not changed relative to the original positions $\measurec_{\dbasepointc} = \measurec_{\dbasepointc^{\prime}}$ and through separate measurable variables that encode holistic data properties, such as orientation or origin.

The compositor function is also expected to be equivariant with respect to changes in data and measurable components $\dfunctc_{\vtotalc}(\vsectionc(\dbasepointc)) = \dfunctc_{\measurec}(\vmarkc(\vsectionc(\dbasepointc)))$ which means that any change to a measurable component input must have a measurably equivalent change in the output. As illustrated in \autoref{fig:artist}, the compositor $\vmarkc$ is expected to assemble the measurable components such that base space changes, e.g., reflection, are reflected in the output; faithfully pass through equivariant measurable components, such as binned colors; and ensure that both types of transformations, here scaling and transposition, are present in the final glyph.

\section{Conclusion}
This paper demonstrates a mathematical formalism for representing the architecture of components in a data visualization library, in which the mathematical structures both enforce well known principles of data visualization and suggest factorizations and decompositions the result in highly modular infrastructure, such that artists can be constructed from composable primitive. This decomposability leads directly to a test framework ambition: if we can split $A=\vmarkc\circ\vchannelc$, and each factor $\vchannelc$, $\vmarkc$ can be further decomposed into composable primitives, we can focus testing on each of these building blocks reducing testing for correctness of the process as a whole to correctness of each building block, and correctness of their chosen combinations. Most of the verification functions in \autoref{sec:artist:equiv} and \autoref{sec:construction:verification} can be directly implemented as assertions in most testing frameworks. A library applying the ideas from \autoref{sec:construction}, created by Thomas Caswell and Kyle Sunden, can be found at \url{https://github.com/matplotlib/data-prototype}. This framework can also be used for discussing API and guiding implementation without strict adoption, as in the architecture discussions of matplotlib's new color mapping pipeline. Further research could investigate building new systems using this model, specifically libraries for visualizing domain specific structured data and domain specific artists. This framework is also well suited for AI driven development in that is a framework for describing visual components in terms of their input, output, and constraints. More research could also explore applying this model to visualizing high dimensional data, particularly building artists that take as input distributed data and artists that are concurrent.
